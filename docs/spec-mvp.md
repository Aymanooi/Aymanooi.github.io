# مواصفات MVP الصارمة: العقود قبل الكود

> **القاعدة:** تعريفات صارمة فقط. **لا كود تنفيذي** (لا `generate.js`، لا منطق). هذا المستند يثبّت العقود.
> **هدف أول نسخة:** ليس 10 صفحات، بل إثبات أن **صفحة واحدة** يمكن أن تكون: **قابلة للاكتشاف + مفيدة عمليًا + قابلة للنمو**.
> **مبدأ حاكم:** لا تُولَّد أي صفحة لا تجتاز "طبقة التحقّق" (Validation Layer). الكمّ ممنوع على حساب القيمة.

---

## 1) الـ schema النهائي لـ `data/opportunities.json`

### 1.1 شكل الملف (Envelope)

```jsonc
{
  "schemaVersion": "1.0.0",
  "archetype": "converter",          // هذا الملف مخصّص لنمط واحد في الـ MVP
  "items": [ /* مصفوفة من Opportunity (انظر 1.2) */ ]
}
```

### 1.2 كائن الفرصة (Opportunity) — الحقول وأنواعها

كل حقل مُعرَّف بـ: النوع، إلزامي/اختياري، القيود.

| الحقل | النوع | إلزامي | القيود / الوصف |
|---|---|---|---|
| `id` | string | ✅ | slug فريد، `^[a-z0-9-]+$`، يُشتق منه المسار `/tools/<id>/` |
| `archetype` | enum | ✅ | `"converter"` فقط في الـ MVP |
| `status` | enum | ✅ | `"draft" \| "approved" \| "rejected"`. **يُولَّد فقط ما حالته `approved`** |
| `category` | enum | ✅ | `"length" \| "weight" \| "temperature" \| "area" \| "volume" \| "data" \| "speed"` |
| `intent` | object | ✅ | **نية البحث الفريدة** (انظر 1.3) — لا تتكرر بين صفحتين |
| `locale` | enum | ✅ | `"ar"` (الـ MVP عربي) |
| `seo` | object | ✅ | `title`, `metaDescription`, `keywords[]` (انظر 1.4) |
| `converter` | object | ✅ | عقد النمط: الوحدتان + المنطق (انظر القسم 2) |
| `content` | object | ✅ | `intro` (≤160 حرف), `howto[]`, `useCases[]`, `faq[]` (انظر 1.5) |
| `related` | string[] | ⬜ | معرّفات `id` لأدوات ذات صلة (يُكمَّل آليًا إن تُرك فارغًا) |
| `discovery` | object | ✅ | بيانات الاكتشاف والترتيب (انظر 1.6) |
| `validation` | object | 🔒 | **يُكتب آليًا** من طبقة التحقّق — لا يُحرَّر يدويًا (انظر القسم 3) |

### 1.3 `intent` — نية البحث الفريدة (جوهر منع التكرار/Doorway)

```jsonc
"intent": {
  "action": "convert",            // enum: convert (في نمط Converter)
  "fromUnit": "meter",            // معرّف وحدة المصدر
  "toUnit": "foot",               // معرّف وحدة الهدف
  "phrase": "تحويل من متر إلى قدم", // العبارة العربية كما يبحث الناس
  "fingerprint": "convert:meter>foot" // بصمة مُعرّفة فريدة (action:from>to)
}
```

- **قاعدة الفرادة:** لا يجوز وجود فرصتين `approved` بنفس `intent.fingerprint`.
- `meter>foot` و`foot>meter` **نيتان مختلفتان** (اتجاه البحث مختلف فعلًا) → صفحتان مشروعتان.
- `meter>foot` و`metre>foot` (مرادف إملائي) **نفس النية** → صفحة واحدة + الباقي canonical/مرادفات داخلية، لا صفحة جديدة.

### 1.4 `seo`

| الحقل | النوع | القيود |
|---|---|---|
| `title` | string | 25–60 حرفًا، يتضمن العبارة، فريد |
| `metaDescription` | string | 120–160 حرفًا، فريد، يتضمن فائدة واضحة |
| `keywords` | string[] | 3–8 عبارات، عربية، لا تتطابق حرفيًا مع صفحة أخرى |

### 1.5 `content` (الحد الأدنى للقيمة الفريدة)

```jsonc
"content": {
  "intro": "حوّل المتر إلى قدم فورًا في متصفحك، بدقة وبدون رفع أي بيانات.",
  "howto": ["أدخل القيمة بالمتر", "اقرأ النتيجة بالقدم فورًا"],   // ≥ 2
  "useCases": ["قياسات البناء", "الرياضة", "السفر"],              // ≥ 3
  "faq": [                                                        // ≥ 3
    { "q": "كم قدم في المتر الواحد؟", "a": "المتر = 3.28084 قدم تقريبًا." }
  ]
}
```

### 1.6 `discovery`

| الحقل | النوع | الوصف |
|---|---|---|
| `source` | enum | `"manual" \| "autocomplete" \| "combinatorial" \| "search-console"` |
| `demandProxy` | number 0–100 | مؤشر طلب تقريبي (عمق autocomplete/اهتمام) |
| `competitionEstimate` | number 0–100 | تقدير منافسة خشن (أقل = أفضل) |
| `arabicGapBonus` | number 0–1 | مضاعِف لفجوة عربية |
| `score` | number | محسوب: `demandProxy × (1 - competition/100) × (1 + arabicGapBonus) × feasible` |
| `feasibleClientSide` | boolean | إن كان `false` → تُرفض حتمًا |
| `discoveredAt` | string (ISO date) | تاريخ الاكتشاف |

## 2) تصميم النمط الوحيد (Converter) — كامل

### 2.1 المدخلات (Inputs)

| الإدخال | النوع | الافتراضي | ملاحظات |
|---|---|---|---|
| `value` | number | `1` | القيمة المراد تحويلها؛ يقبل العشري والسالب (حسب الوحدة) |
| `direction` | enum | `forward` | `forward` = from→to، `reverse` = to→from (زر تبديل اختياري في الواجهة) |

> الصفحة الواحدة تثبّت `fromUnit/toUnit` من البيانات؛ المستخدم يغيّر `value` فقط (و`direction` إن فُعّل).

### 2.2 المخرجات (Outputs)

| المخرج | الوصف | لماذا يضيف قيمة فريدة |
|---|---|---|
| `result` | الرقم المحوّل، مُنسّق بدقة `precision` | جوهر الأداة |
| `formatted` | نتيجة بصيغة مقروءة + رمز الوحدة | تجربة أوضح |
| `referenceTable` | جدول قيم شائعة (1،2،5،10،50،100…) | **محتوى فريد عملي** يميّز الصفحة عن منافس فقير |
| `formula` | عرض المعادلة المستخدمة (شفافية) | ثقة + قيمة تعليمية |

### 2.3 عقد المنطق (`converter` object)

```jsonc
"converter": {
  "dimension": "length",            // البُعد الفيزيائي (تحقّق التطابق)
  "from": { "id": "meter", "symbol": "م", "name": "متر" },
  "to":   { "id": "foot",  "symbol": "قدم", "name": "قدم" },
  "logic": {
    "kind": "linear",               // enum: "linear" | "affine" | "formula"
    "factor": 3.28084,              // linear/affine: result = value*factor (+offset)
    "offset": 0,                    // affine فقط (مثل الحرارة)
    "formulaId": null               // formula فقط: مرجع لدالة نقية في /lib/converters.js
  },
  "precision": 4,                   // عدد المنازل العشرية للعرض
  "referenceValues": [1, 2, 5, 10, 50, 100]
}
```

**أنواع المنطق الثلاثة (تغطي كل المحوّلات):**

| `kind` | المعادلة | أمثلة |
|---|---|---|
| `linear` | `result = value × factor` | طول، وزن، مساحة، حجم، بيانات |
| `affine` | `result = value × factor + offset` | الحرارة (°م→°ف: ×9/5 +32) |
| `formula` | دالة نقية مرجعية بـ `formulaId` | حالات خاصة نادرة |

> **مبدأ العزل:** كل المنطق دوال نقية في `/lib/converters.js` (مُختبَرة). ملف الصفحة يربط الإدخال بالمنطق بالـ DOM فقط — لا حساب في الـ HTML.

### 2.4 ضمان الصحّة (Correctness)

كل وحدة في النظام لها **قيمة مرجعية موثّقة** تُختبَر آليًا:
- مثال إلزامي: `1 meter → 3.28084 foot (±1e-4)`.
- لا تُعتمد فرصة تحويل إلا إذا مرّ اختبار قيمتها المرجعية في `/tests`.

## 3) طبقة التحقّق الإلزامية (Validation Layer) — بوّابة ما قبل التوليد

> تعمل وقت البناء **قبل** المولّد. أي فرصة تكسر قاعدة `blocker` → `status = "rejected"` و**لا تُولَّد صفحتها**. النتائج تُكتب في `opportunity.validation`.

| المعرّف | القاعدة | الشدّة | كيف يُتحقّق منها |
|---|---|---|---|
| **U1 — حساب حقيقي** | التحويل غير تافه: `from.id ≠ to.id`، و`factor ≠ 1` (إلا بمبرّر)، ومنطق مُعرّف | 🔴 blocker | فحص بنيوي للحقول |
| **U2 — نية فريدة** | `intent.fingerprint` لا يتكرر بين المعتمدة | 🔴 blocker | فهرسة البصمات + كشف التصادم |
| **U3 — قابلية client-side** | `discovery.feasibleClientSide = true` | 🔴 blocker | فحص العلم |
| **U4 — كفاية المحتوى** | `faq ≥ 3` و`useCases ≥ 3` و`howto ≥ 2` و`referenceValues ≥ 5` و(محتوى مولّد ≥ 250 كلمة) | 🔴 blocker | عدّ الحقول/الكلمات |
| **U5 — إشارة طلب** | `demandProxy ≥ 20` أو `source = "manual"` (تجاوز موثّق) | 🟡 warn→blocker | عتبة رقمية |
| **U6 — لا تكرار SEO** | `title`/`metaDescription`/`keywords` فريدة (لا تطابق صفحة أخرى) | 🔴 blocker | مقارنة تطبيع نصّي |
| **U7 — صحّة الحساب** | القيمة المرجعية للوحدة مرّت في الاختبارات | 🔴 blocker | ربط بنتائج `/tests` |
| **U8 — جودة SEO شكلية** | أطوال `title` (25–60) و`description` (120–160) ضمن الحدود | 🟡 warn | قياس الأطوال |
| **U9 — اكتمال الحقول** | كل الحقول الإلزامية في الـ schema موجودة وصحيحة النوع | 🔴 blocker | تحقّق schema |

**سجلّ النتيجة في الكائن:**

```jsonc
"validation": {
  "passed": true,
  "checkedAt": "2026-06-30T00:00:00Z",
  "blockers": [],            // قائمة معرّفات القواعد المكسورة (blocker)
  "warnings": ["U8"],        // تحذيرات لا تمنع التوليد
  "usefulnessScore": 87      // درجة فائدة مجمّعة (شفافية)
}
```

> **قاعدة ذهبية:** `passed = (blockers.length === 0)`. التوليد يقرأ فقط `status="approved" && validation.passed=true`.

## 4) قواعد SEO الصارمة (مكافحة Doorway/التكرار)

| القاعدة | التعريف الإجرائي |
|---|---|
| **نية مختلفة فعلًا لكل صفحة** | كل صفحة = `intent.fingerprint` فريد = زوج (action، from>to) مختلف. لا صفحتان لنفس النية |
| **لا محتوى مكرّر** | `intro`/FAQ/جدول لكل صفحة مشتقّ من وحداتها وقيمها → نص مختلف فعليًا، لا قالب فارغ مكرّر |
| **canonical لكل صفحة** | المرادفات الإملائية تُوجَّه canonical للصفحة الأساس، لا تُنشأ صفحات لها |
| **قيمة قائمة بذاتها** | كل صفحة تعمل وتفيد دون الاعتماد على صفحة أخرى (ليست جسرًا/بوابة) |
| **بيانات منظّمة حقيقية** | JSON-LD يصف أداة فعلية (`WebApplication`) + FAQ حقيقية، لا حشو |

## 5) هدف أول نسخة: صفحة واحدة (لا 10)

**النسخة الأولى = صفحة `convert:meter>foot` فقط**، تثبت الثلاثية:

| المعيار | إثباته في الصفحة الواحدة |
|---|---|
| **قابلة للاكتشاف (Discoverable)** | URL نظيف + title/description فريدة + JSON-LD + إدراج في `sitemap.xml` + canonical |
| **مفيدة عمليًا (Useful)** | أداة تحويل تعمل فورًا في المتصفح + جدول قيم مرجعي + اجتازت U1–U9 |
| **قابلة للنمو (Growable)** | مولّدة من `opportunity` واحد عبر القالب-النمطي → الصفحة الثانية = إضافة `item` واحد. صفر إعادة هيكلة |

**معايير القبول (Acceptance) لأول نسخة:**
1. الصفحة مُولَّدة من بيانات (لا HTML مكتوب يدويًا).
2. اجتازت طبقة التحقّق (`validation.passed = true`).
3. اختبار صحّة الحساب يمر (`1 m = 3.28084 ft`).
4. Lighthouse: أداء ≥ 95، SEO ≥ 95.
5. إضافة `item` ثانٍ ينتج صفحة ثانية **دون تعديل أي كود** → إثبات النمو.

---

## 6) ما الذي يتبقّى تعريفه قبل الكود؟ (لا شيء حاسم)

العقود الثلاثة مثبّتة الآن: **schema** (القسم 1) + **archetype** (القسم 2) + **Validation Layer** (القسم 3) + قواعد SEO (4) + هدف الصفحة الواحدة (5). عند الاعتماد، يبدأ التنفيذ بهذا الترتيب الصارم:

```
1) lib/converters.js (دوال نقية) + اختباراتها (U7)
2) ملف بيانات opportunities.json بعنصر واحد (meter>foot)
3) قالب converter + المولّد (يقرأ المعتمد فقط)
4) طبقة التحقّق (تشتغل قبل المولّد)
5) sitemap + Actions + قياس Lighthouse
```

> المستند حيّ؛ يُحدَّث إن تغيّرت الأدلة.
