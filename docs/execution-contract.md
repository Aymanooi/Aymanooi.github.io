# عقد التنفيذ النهائي (قبل الكود) — منع الانحراف

> آخر تثبيت قبل أمر التنفيذ. كل بند قرار قاطع لا يتغيّر أثناء البناء. **لا كود في هذا المستند.**

---

## 1) واجهة الإدخال/الإخراج في `lib/converters.js`

**القرار: طبقتان فقط — أساس عام + غلاف مُسمّى. قابل للتوسّع بلا over-engineering.**

| الدالة | التوقيع | السلوك |
|---|---|---|
| `convertLinear(value, factor)` | `(number, number) → number` | ترجع `value * factor` خامًا (دقة كاملة). هذا **مفصل التوسّع**: الصفحة 2 (كجم→رطل) تعيد استخدامها بـ`factor` آخر |
| `metersToFeet(value)` | `(number) → number` | غلاف رفيع = `convertLinear(value, 3.28084)` |

**لماذا هذا "قابل للتوسّع بلا إفراط":**
- أساس عام واحد (`convertLinear`) + اسم واضح واحد (`metersToFeet`). **لا سجلّ وحدات، لا كائنات إعداد، لا factory.** فقط ما يكفي ليُعاد استخدام الأساس لاحقًا.
- **مدخل غير صالح:** `value` يُمرَّر عبر `Number()`؛ إن كان الناتج `غير منتهٍ` (NaN/Infinity) → الدالة ترجع `NaN` (سلوك معرّف ومُختبَر).
- **الإخراج خام (raw float).** التقريب/التنسيق **ليس** مسؤولية المنطق — يحدث في طبقة العرض. هذا يبقي المنطق نقيًا قابلًا للاختبار.

## 2) اختبار النجاح: المقارنة والإبلاغ

**القرار: مقارنة بـ tolerance (ليست strict equality)، عبر `assert.ok` مع مساعد محلي صغير داخل ملف الاختبار فقط.**

- **لماذا ليست `===`:** نتعامل مع أعداد عائمة (floats)؛ المساواة الصارمة هشّة. نستخدم:
  `Math.abs(actual - expected) < 1e-4`.
- **الآلية:** مساعد محلي `assertClose(actual, expected, tol = 1e-4, msg)` **داخل ملف الاختبار حصرًا** (ليس مكتبة، ليس إطارًا) يلفّ `assert.ok(...)` من `node:assert/strict`.
- **حالات الاختبار:**
  | الإدخال | المتوقّع |
  |---|---|
  | `metersToFeet(1)` | `≈ 3.28084` |
  | `metersToFeet(0)` | `0` |
  | `metersToFeet(2.5)` | `≈ 8.2021` |
  | `metersToFeet(-3)` | `≈ -9.84252` |
  | `metersToFeet("abc")` | `NaN` (عبر `Number.isNaN`) |
  | `convertLinear(10, 2)` | `20` (الأساس العام) |
- **أين يظهر الفشل عمليًا:** في **الطرفية / CI** عبر `node --test` — مخرجات TAP تطبع `not ok` مع رسالة `assertClose` (المتوقّع مقابل الفعلي)، ورمز خروج غير صفري. **لا يظهر في المتصفح** — الاختبار طبقة بناء لا طبقة تشغيل.

## 3) تدفّق execution داخل المتصفح (تسلسل فعلي)

**القرار: استيراد وحدة واحدة → حساب أوّلي عند التحميل → تحديث بالطفرة المباشرة (direct mutation)، بلا re-render.**

```
1. المتصفح يحلّل HTML من الأعلى. <link rel="stylesheet"> في <head> يُطبَّق.
2. في نهاية <body>: <script type="module"> — الوحدات مؤجَّلة افتراضيًا (defer)،
   تُنفَّذ بعد اكتمال تحليل DOM.
3. الاستيراد أولًا:  import { metersToFeet } from "../../lib/converters.js"
   → المتصفح يجلب ملف الوحدة النقي (ملف ثابت واحد).
4. عند تنفيذ الوحدة:
   - الحصول على المراجع مرة واحدة:
       input  = document.getElementById("value")
       output = document.getElementById("result")
   - تعريف update():
       n = Number(input.value)
       r = metersToFeet(n)
       output.textContent = Number.isNaN(r) ? "—" : r.toFixed(4) + " قدم"
5. حساب أوّلي:  update()  ← يُستدعى مرة عند التحميل (الحقل مُعبّأ بـ 1 → يعرض 3.2808 قدم)
6. ربط الحدث:  input.addEventListener("input", update)
7. عند كتابة المستخدم: حدث input → update() →
   **طفرة مباشرة لعقدة نصّية واحدة** (output.textContent = ...).
   لا re-render، لا innerHTML، لا virtual DOM، لا إعادة بناء للصفحة.
```

- **re-render أم direct mutation؟** → **direct mutation** لعقدة نصّ واحدة. أرخص وأسرع تحديث ممكن.
- **التنسيق:** `toFixed(4)` في طبقة العرض فقط (المنطق يبقى خامًا).
- **صفر طلبات شبكة** بعد تحميل الصفحة؛ تعمل بلا اتصال.

---

## 4) منع الانحراف: لا abstraction إضافي في الواجهة

**القرار: مفروض وصارم لهذه المرحلة.**

| ممنوع الآن | مسموح |
|---|---|
| ❌ أي framework (React/Vue/...) | ✅ DOM أصلي (vanilla) فقط |
| ❌ أي خطوة بناء/bundler | ✅ ES module مستورد مباشرة |
| ❌ ملف `helpers.js` أو طبقة أدوات | ✅ المنطق النقي في `lib/` + غراء مضمّن في الصفحة |
| ❌ أي اعتماد خارجي (npm runtime) | ✅ المتصفح + `node:test` للاختبار فقط |
| ❌ تعميم مبكّر (سجلّ وحدات، إعدادات) | ✅ `convertLinear` + `metersToFeet` فقط |

> الاستيراد الوحيد في الصفحة هو الدالة النقية. السكربت المضمّن **غراء يربط الإدخال بالمنطق بالـDOM** — لا أكثر.

## 5) تعريف "تم النجاح" — جملة واحدة قابلة للتحقّق

> **عند فتح `tools/meter-to-foot/index.html`، تعرض الصفحة فورًا «1 متر = 3.2808 قدم» وتُحدّث الناتج صحيحًا لأي إدخال رقمي، و`node --test` يمر دون فشل — والصحّة الحسابية هي المقياس الوحيد لهذه المرحلة.**

(زمن التحميل الفوري خاصيةٌ بنيوية لصفحة ثابتة، لا بوّابة مُقاسة الآن — تبقى مُجمّدة كمقياس وفق نقطة التجميد.)

---

## 6) ترتيب التنفيذ عند الـ go
```
1) lib/converters.js          → convertLinear + metersToFeet
2) tests/converters.test.js   → assertClose + الحالات الستّ → node --test يمر
3) tools/meter-to-foot/index.html → الصفحة + غراء الاستيراد
4) تحقّق يدوي في المتصفح + commit/push للفرع
```

> هذا العقد مُقفل. أي تغيير عليه يتطلّب قرارًا صريحًا، لا اجتهادًا أثناء الكتابة.

## 7) مراجعة ما قبل التنفيذ (تعديل معتمد قبل الـ go)

ثلاث إضافات صغيرة تضمن أن ما نبنيه **أصل قابل للاستمرار**، لا مجرد كود صحيح. لا توسعة ولا تعقيد.

### 7.1 تأطير صريح: هذه شريحة اختبار فرضية، لا "نواة النظام"
> لا يوجد أي افتراض ضمني بأن هذا التصميم الحالي هو نواة النظام الكامل.
> صفحة `meter→foot` هي **أول اختبار حقيقي للفرضية** (أن أداة عربية بسيطة client-side تُكتشَف وتُستخدَم)،
> وليست بداية بناء نظام مكتمل. المولّد/التحقّق/الاكتشاف تبقى **مُجمّدة** حتى تثبت الفرضية.

### 7.2 معيار الإيقاف (Kill Switch) — لا توسّع تلقائي بلا دليل
> **لا تُبنى أي صفحة ثانية تلقائيًا.** التوسّع قرار صريح مشروط بدليل من الصفحة الأولى.

| الشرط بعد النشر والفهرسة | القرار |
|---|---|
| خلال نافذة 60–90 يومًا: ~صفر انطباعات عضوية رغم الفهرسة | **إيقاف** هذا الاتجاه وإعادة تقييم (pivot)، لا توسّع |
| فشل اختبار الاستخدام البشري (7.3) | **إيقاف** حتى يُصلَح، قبل أي توسّع |
| ظهور انطباعات/استخدام حقيقي ينمو | **عندها فقط** يُفكّ تجميد الصفحة الثانية |

> الافتراضي = **التوقّف**، لا الاستمرار. الاستمرار يتطلّب إثباتًا، لا العكس.

### 7.3 تعريف النجاح الحقيقي (بشري، يُضاف للعقد)
النجاح **ليس** "اختبار يمر" فقط. يُضاف معيار قبول بشري:
> **مستخدم حقيقي يصل للأداة ويفهمها ويستخدمها بلا شرح خلال أقل من 10 ثوانٍ.**

- **هذا الآن جزء من العقد** كـ"بوّابة قبول بشرية"، **بجانب** الصحّة الحسابية.
- التمييز الصادق: الصحّة الحسابية = **المقياس الآلي** (pass/fail عبر `node --test`)؛
  قاعدة الـ10 ثوانٍ = **بوّابة قبول بشرية** (حكم نوعي على الوضوح والبساطة).
- كلاهما رخيص التحقّق لصفحة واحدة، ولا يضيف أي تجريد أو بنية.

### 7.4 تحديث جملة النجاح (تَجُبّ ما في القسم 5)
> **النجاح = (1) `node --test` يمر [صحّة آلية] + (2) مستخدم حقيقي يفهم الأداة ويستخدمها بلا شرح خلال < 10 ثوانٍ [قبول بشري].** الصفحة تعمل بلا اتصال وبلا طلبات خادم كخاصية بنيوية.

## 8) القفل الحاسم الأخير (Critical Lock) — بروتوكولات قابلة للتحقّق

### 8.1 بروتوكول قياس "10 ثوانٍ فهم واستخدام"

> ليس عبارة عامة، بل **سيناريو واحد + معيار فهم مرصود + حد أدنى للتفاعل**.

**سيناريو الاختبار (واحد فقط):**
- مستخدم عربي لم يرَ الصفحة من قبل، على الجوال.
- يُعطى **المهمة فقط** (لا شرح للأداة): *«كم قدمًا في 5 أمتار؟ استخدم هذه الصفحة.»*
- يُسلَّم الرابط، ويبدأ المؤقّت عند ظهور الصفحة.

**كيف نقرر أنه "فهم الأداة" (سلوك مرصود، بلا أسئلة موجِّهة):**
- يحدّد حقل الإدخال، ويغيّر القيمة إلى `5`، ويقرأ الناتج — **دون أن يسأل "أين أكتب؟"** ودون أن يحتاج قراءة المحتوى الشارح أولًا.

**الحد الأدنى من التفاعل الذي يُعدّ نجاحًا:**
- خلال **< 10 ثوانٍ**: (أ) غيّر الإدخال إلى `5` **و** (ب) أشار/نطق الناتج الصحيح (`16.4042 قدم`).
- أي طلب تلميح ("وين أكتب؟") أو حيرة > 10 ثوانٍ = **فشل**.

**عتبة القبول:** **مستخدمان اثنان مستقلّان** (naive) ينجحان كلاهما بلا مساعدة < 10 ثوانٍ.
> صدق: هذه فحص استخدام نقطي (spot-check) لا دراسة إحصائية — مناسب لصفحة واحدة، وثنائي القيمة (نجح/فشل) قابل للرصد.

### 8.2 تعريف "فشل الصفحة" بالأرقام (شجرة قرار صارمة)

> الأداة القياسية = **Google Search Console** (مجاني). نفرّق بدقّة بين ثلاث حالات حتى لا نخلط العلاج:

| الحالة المرصودة | الإشارة | الحد الرقمي | القرار |
|---|---|---|---|
| **غير مفهرسة** | لا تظهر في `site:` / Coverage | لم تُفهرس خلال **30 يومًا** | عطل تقني/SEO → **يُصلَح**، ليس pivot |
| **مفهرسة بلا طلب** | Impressions عضوية | **< 50 انطباعًا خلال 90 يومًا** رغم الفهرسة | فرضية الطلب **مكذّبة** → **إيقاف/pivot** |
| **مفهرسة بطلب بلا نقر** | CTR | **≥ 200 انطباعًا و~0 نقرة** | اكتشاف نعم، إقناع لا → **يُصلَح العنوان/الوصف**، ليس pivot |

> "الفشل" الذي يُفعّل الـKill Switch = **الحالة الوسطى تحديدًا** (مفهرسة + < 50 انطباعًا/90 يومًا). الأرقام عتبات بدء قابلة للمعايرة بالأدلة، لا مطلقات.

### 8.3 منع الانحراف المستقبلي — تأكيد صريح

> **نعم. يُمنع في هذه المرحلة أي ميزة لا تخدم مباشرة: الحساب + العرض + الاختبار + (سكافولد الاكتشاف المقفل أصلًا).**

**النطاق المسموح = ما هو مقفل في هذا العقد حصرًا:**
- منطق الحساب النقي (`lib`), غراء العرض في الـDOM, الاختبار الآلي,
- ووسوم SEO للصفحة + جدول مرجعي واحد + كتلة محتوى/FAQ واحدة (جزء من فرضية الاكتشاف، **ليست تحسينات**).

**ممنوع منعًا صريحًا (يتطلّب قرارًا مستقلًا، لا اجتهادًا):**
- ❌ زر نسخ، زر مشاركة، وضع ليلي/تبديل، رسوم متحركة، أيقونات إضافية.
- ❌ تحليلات (analytics)، PWA، service worker، تعدّد لغات.
- ❌ وحدات إضافية، اتجاه عكسي، "أدوات ذات صلة" تفاعلية.
- ❌ أي مولّد/تحقّق/اكتشاف (مُجمّد أصلًا).

> القاعدة: إن لم تخدم الميزة **الحساب أو العرض أو الاختبار أو الاكتشاف المقفل** — فهي مؤجّلة. لا يتحوّل المشروع إلى نظام أكبر من اللازم عبر إضافات "تحسينية" متسلّلة.

## 9) اختبار الإجهاد الواقعي الأخير (Reality Stress Test) — قفل نهائي

### 9.1 حجم عيّنة اختبار "10 ثوانٍ" (تصحيح خطر الـ false positive)

> القرار المُختار: **5 مستخدمين مستقلّين** (لا "مرتان لكل مستخدم").

- **لماذا 5 مستخدمين وليس إعادة الاختبار مرتين:** إعادة الاختبار لنفس المستخدم **ملوّثة بأثر التعلّم** — بعد المحاولة الأولى يعرف مكان الحقل، فالمحاولة الثانية **ليست مستقلّة** وتعطي تفاؤلًا زائفًا. 5 مستخدمين مستقلّين يقطعون أثر التعلّم ويغطّون تباين الأجهزة/الأعمار.
- **عتبة القبول المُحدّثة:** **4 من 5 ينجحون** (< 10 ثوانٍ، بلا مساعدة). يسمح بشذوذ واحد دون أن يكسر الحكم.
- يَجُبّ هذا عتبة "مستخدمان" في 8.1.
> صدق: 5 ليست عيّنة إحصائية صارمة، لكنها في أبحاث الاستخدام (Nielsen) تكشف ~85% من مشكلات القابلية للاستخدام — كافية وعملية لصفحة واحدة.

### 9.2 Kill Switch — شرط مزدوج يفصل "غياب الطلب" عن "سوء تنفيذ SEO"

> مخاوفك محقّة: الاعتماد على Impressions وحده قد يُحمّل "الطلب" وزر مشكلةٍ مصدرها الفهرسة/الترتيب.
> **التصحيح المنطقي:** الشرط المزدوج الصحيح **ليس** "impressions منخفضة + CTR منخفضة" (الـCTR بلا معنى عند ~0 انطباع)، بل:

**لا يُعلَن "فشل طلب" ويُفعَّل الـpivot إلا بتحقّق *شرطين معًا*:**
1. **تنفيذ SEO سليم مؤكَّد** (يستبعد أن المشكلة فهرسة/ترتيب): الصفحة **مفهرسة** (Coverage/`site:`) **و** تظهر فعلًا لاستعلامها المستهدف (لها "متوسط ترتيب" في Search Console = Google يعرضها).
2. **الطلب غائب رغم ذلك:** **< 50 انطباعًا خلال 90 يومًا**.

| إن تحقّق | الاستنتاج | القرار |
|---|---|---|
| (1) فشل: غير مفهرسة/لا تظهر | المشكلة **تنفيذ SEO** | **يُصلَح** التنفيذ — **ممنوع** pivot |
| (1) نجح + (2) تحقّق | الطلب **غائب فعلًا** | **pivot** (الـKill Switch) |
| (1) نجح + انطباعات جيدة + ~0 نقرة (≥200 انطباع) | إقناع/وصف ضعيف | يُصلَح العنوان/الوصف — ليس pivot |

> الخلاصة: **شرط مزدوج** (SEO سليم **و** طلب غائب) — لا نقتل الفكرة بسبب خطأ تنفيذي قابل للإصلاح.

### 9.3 نطاق منع الإضافات: مرحلي لا دائم (تأكيد صريح)

> **المنع مؤقّت لمرحلة الـMVP فقط، وليس قاعدة دائمة للنظام.**

- النظام **قابل للتطوّر بالتصميم**: بنود NICE-TO-HAVE والأنماط والمولّد كلها **مُجمّدة لا محذوفة**، تُفكّ **بقرار صريح** بعد اجتياز الشريحة الأولى لبواباتها.
- المنع الحالي = **انضباط مؤقّت لإثبات الفرضية أولًا**، لا إغلاق على نموذج واحد.
- المعمارية مفتوحة وممتدة (طبقات + أنماط)؛ ما يتغيّر بعد النجاح هو **رفع التجميد تدريجيًا بالأدلة**، لا إعادة هيكلة.

## 10) حالة الجاهزية النهائية

كل الزوايا مُغلقة: الواجهات (3) · الاختبار (8.1) · الفشل (8.2) · منع الانحراف (8.3) · الفرضية/الإيقاف (7) · إجهاد الواقع (9: عيّنة 5، شرط مزدوج، منع مرحلي).
**الحالة: Zero-Ambiguity Execution State.** التنفيذ ينتظر أمر `go` الصريح فقط.

## 11) بوّابة الحقيقة الواحدة (Single Truth Gate) — قفل أخير

### 11.1 فصل "صحة الحساب" عن "صحة المنتج" — الإجابة الصريحة

> **لا. الحساب وحده غير كافٍ. الحد الأدنى من UX شرط قبول إلزامي.**

تسوية التناقض الظاهري: "الصحّة الحسابية = المقياس الوحيد" تعني **المقياس الآلي** (pass/fail عبر `node --test`).
أما **صحة المنتج** فتتطلّب طبقتين معًا:

| الطبقة | ما تقيس | أداة التحقّق | متى |
|---|---|---|---|
| صحّة الحساب | هل الرقم صحيح؟ | `node --test` | وقت البناء |
| **حد أدنى UX (بناء)** | هل الأداة واضحة وقابلة للاستخدام فورًا؟ | **قائمة فحص بناء** (أدناه) أراجعها بنفسي قبل الحفظ | وقت البناء |
| قبول بشري ميداني | هل يستخدمها غريب < 10ث؟ | 5 مستخدمين (9.1) | بعد النشر |

**قائمة فحص الحد الأدنى من UX (بوّابة بناء — يجب أن تتحقّق كلها قبل commit):**
- ✅ حقل الإدخال ظاهر **فوق الطية** بلا تمرير.
- ✅ النتيجة ظاهرة بجانب/تحت الإدخال مباشرة، بخط كبير واضح.
- ✅ مثال مُعبّأ مسبقًا (`1` → `3.2808 قدم`) يُظهر أن الأداة "تعمل" دون أي إجراء.
- ✅ تسمية عربية واضحة للإدخال والنتيجة (متر / قدم).
- ✅ تحديث حيّ فوري عند الكتابة (بلا زر "احسب").
- ✅ RTL سليم، تباين كافٍ، يعمل على عرض جوال.

> صفحة صحيحة حسابيًا لكنها تكسر أي بند أعلاه = **مرفوضة**، لا تُحفظ.

### 11.2 خطوة "تحقّق الوصول" قبل التنفيذ — أُجريت فعلًا

> سؤالك: هل نحتاج تحقّق وصول بسيط قبل البناء؟ **نعم، وأجريناه — بنتيجة إيجابية.**

**تفريق صادق بين ما يُتحقّق منه الآن وما هو مستحيل قبل النشر:**
| الطبقة | قابلة للتحقّق قبل البناء؟ | النتيجة |
|---|---|---|
| **قناة التوصيل** (هل Pages يخدم؟) | ✅ نعم | `https://aymanooi.github.io` → **HTTP 200** (مُثبَت). القناة حيّة |
| وصول مستخدم حقيقي للصفحة | ❌ لا (تحتاج صفحة منشورة) | يُقاس بعد النشر |
| عرض Google لها | ❌ لا (مستحيل قبل وجود الصفحة) | بوّابة الفهرسة 30 يومًا (8.2) |

> لا نُنظّر: قناة النشر **مُختبَرة وتعمل**. أما "هل يصل المستخدم" و"هل يعرضها Google" فمستحيل إثباتهما قبل وجود الصفحة — وهما في بوّابات ما بعد النشر (الفهرسة + الـKill Switch)، لا قبله.

**خطوة ما بعد النشر الإلزامية المضافة:** بمجرد نشر الصفحة → تقديم رابطها في Google Search Console (URL Inspection) لتسريع الفهرسة، وبدء عدّاد الـ30/90 يومًا.

## 13) تعريف "قابلية التوسّع الناجحة" (Scalability Success) — قفل ما قبل التنفيذ

### 13.1 الاعتراف الصادق أولًا
> **لا يمكن إثبات "قابلية التكرار" من مثالٍ واحد.** التجريد يُثبَت بالاستخدام الثاني والثالث لا الأول (قاعدة الثلاثة).
> لذلك لا أدّعي أن الصفحة #1 *تُثبت* التوسّع؛ بل أُلزم الصفحة #1 بـ**ثوابت فصل** تجعل اختبار التوسّع *قابلًا للنجاح*، وأعرّف **الاختبار نفسه** بدقّة ليُطبَّق عند الصفحة #2.

### 13.2 ما الذي يمنع أن تكون الصفحة #1 نجاحًا معزولًا؟ — ثوابت التكرار (تُبنى داخل #1)

| الثابت | الإلزام في الصفحة #1 | ما يضمنه |
|---|---|---|
| **فصل المنطق عن البيانات** | كل خاص-بالمثيل (الوحدتان، factor، precision، نصوص SEO، FAQ، القيم المرجعية) في **كائن بيانات واحد** أعلى الصفحة. `convertLinear` في `lib` **عام تمامًا، صفر خصوصية** | تبديل كائن البيانات = صفحة محوّل أخرى صحيحة بلا أي تعديل آخر |
| **ثبات القالب** | هيكل HTML لا يحوي أي افتراض عن "متر/قدم" إلا عبر كائن البيانات | الصفحة #2 = نفس الهيكل، تتغيّر البيانات فقط |
| **UX على مستوى النمط لا الصفحة** | قائمة فحص UX (11.1) تنطبق كما هي على أي محوّل | لا إعادة تصميم UX لكل صفحة |

> هذه الثوابت هي **ما يمنع النجاح المعزول**: إن تحقّقت، الصفحة #2 آلية ميكانيكيًا لا إبداعية.

### 13.3 التعريف العملي القابل للاختبار (بوّابة التوسّع — تُطبَّق عند الصفحة #2)

> **"قابلية توسّع ناجحة" = إنتاج الصفحة #2 يتطلّب تغيير *كائن البيانات فقط*، بحيث:**
> 1. **صفر منطق جديد** في `lib` (أو على الأكثر غلاف مُسمّى من سطر واحد يستدعي `convertLinear`).
> 2. **صفر تغيير بنيوي** في القالب (HTML skeleton).
> 3. **صفر تغيير في CSS**.
> 4. الصفحة #2 تمرّ **نفس** اختبار الصحّة الآلي + **نفس** قائمة فحص UX دون تعديلهما.
>
> **معيار الفشل (يكسر النظام):** إن احتاجت #2 أي إعادة هيكلة منطق/قالب/UX → **التوسّع مكذّب** → يُعاد تصميم النمط **قبل** الصفحة #3 (لا نتراكم على أساس مكسور).

### 13.4 الحد الأدنى على الصفحة #1 الذي يمنحنا الثقة

عمليًا، نُعلن "ثقة في التوسّع" إذا تحقّق في #1:
- ✅ كل الخصوصيات معزولة في **كائن بيانات واحد** (لا قيمة وحدة مبعثرة في الـmarkup/المنطق).
- ✅ `lib` يحوي `convertLinear` العام + غلاف `metersToFeet` رفيع فقط (لا منطق خاص بالطول).
- ✅ **اختبار الاستبدال الذهني (dry-run):** استبدال كائن البيانات بـ(كجم→رطل، factor=2.20462) **على الورق** يُنتج صفحة صحيحة بلا أي تعديل بنيوي. إن نجح ذهنيًا → الثوابت سليمة.

> النمط Converter يتوقّع أصلًا ثلاثة أنواع (`linear`/`affine`/`formula`)، فالحرارة (affine) **ليست مفاجأة** بل حالة مُتوقَّعة في العقد — ما يقلّل خطر "انهيار عند نوع جديد".

## 14) قاعدة القرار النهائي (مطابقة لشرطك)

## 15) النموذج المُختار: تطوّر مضبوط (Evolvable Architecture) — إغلاق الثغرة المنطقية

### 15.1 الخيار
> # **B — تطوّر مضبوط.** (وليس A الثبات الصارم.)

**لماذا B حتمًا:** قالب مبنيّ على مثال واحد **ليس نهائيًا بالضرورة**. فرض ثباته (A) = **تجميد تجريد سابق لأوانه** — وهو بالضبط الخطأ الذي يحوّل "الالتزام بالقالب" إلى قيد يمنع التحسين. A سريع البداية، قاتل التطوّر.

### 15.2 الإجابة على سؤالك الحاسم: تغيير القالب = تطوّر طبيعي أم فشل؟

> **التمييز الجوهري الذي كان ناقصًا: ليس "تغيير القالب" مقابل "ثباته"، بل "تطوّر مشترك" مقابل "تشظٍّ لكل صفحة".**

| الحالة | الحكم |
|---|---|
| القالب المشترك يتطوّر (v1→v2)، وكل الصفحات **تُعاد توليدها** منه فتبقى متّسقة | ✅ **تطوّر طبيعي** — مرغوب، ليس فشلًا |
| الصفحة #2 تحتاج قالبًا/منطقًا **خاصًّا بها** يتفرّع عن البقية (fork) | ❌ **فشل النظام** (تشظٍّ) — هذا ما نمنعه |

> إذًا اكتشاف أن القالب يحتاج تعديلًا بعد #2 أو #3 = **تطوّر طبيعي**، **شرط** أن يكون التعديل على القالب **المشترك الواحد** ويسري على الجميع. الفشل ليس "تغيّر القالب"، بل "**تفرّعت الصفحات**".

### 15.3 قواعد التطوّر المضبوط (حتى لا يصبح B فوضى)

1. **مصدر واحد مشترك:** قالب واحد + schema واحد. **لا صفحة تملك نسخة خاصة (no per-page fork).**
2. **التغيير يسري على الكل:** أي تعديل للقالب **يرفع نسخته** (`templateVersion`) و**يُعاد توليد كل الصفحات** منه → الاتساق ثابت دائمًا.
3. **مدفوع بالأدلة:** تعديل القالب يُبرَّر باكتشاف حقيقي (نتيجة UX، نوع archetype جديد)، لا بالهوى.
4. **توافق البيانات للخلف:** تغييرات الـschema **إضافية أو مُرحَّلة (migrated)**, لا كاسرة صامتة للبيانات القائمة.
5. **إعادة تعريف الفشل:** النظام يفشل فقط حين تحتاج صفحة شيئًا **يستحيل التعبير عنه** في القالب/الschema المشترك → يفرض fork. عندها: إمّا **نوسّع القالب المشترك** (تطوّر)، أو نُقرّ أنه **نمط مختلف** (archetype جديد).

### 15.4 تصحيح تعريف 13.3
> "توسّع ناجح" **لا يعني** "القالب لا يتغيّر أبدًا". يعني:
> **(أ)** الحالة الشائعة: صفحة جديدة = تغيير بيانات فقط على القالب المشترك القائم؛
> **(ب)** عند الحاجة: القالب المشترك **يتطوّر بنسخة واحدة تسري على الجميع** — لا تفرّع لكل صفحة.
> الثابت الحقيقي ليس "القالب جامد"، بل **"قالب واحد مشترك، صفر تشظٍّ"**.

## 17) نظام حوكمة تغيير القالب (من يقرّر + الحد الأدنى)

### 17.1 مصدر قرار تغيير القالب — الإجابة: **مزيج مُرجَّح حسب نوع التغيير**

> ليس مصدرًا واحدًا، لأن كل مصدر **يحكم طبقة مختلفة**. والمطوّر **مُنفّذ لا سلطة مستقلّة**: لا يغيّر القالب بالذوق، بل فقط حين يعبر دليلٌ عتبته.

| نوع التغيير | المصدر صاحب السلطة | العتبة المُلزِمة |
|---|---|---|
| صحّة/قدرة (لا يمكن التعبير عنها في القالب) | **الاختبارات الآلية** (موضوعي) | فشل تعبيري مُثبَت → توسيع القالب |
| فهم/استخدام (UX) | **اختبار المستخدم** (5 مستخدمين) | **≥ 2 من 5 يفشلون لنفس السبب** (عيب قابل للتكرار) |
| اكتشاف/إقناع (ترتيب) | **بيانات Search Console** | فرع الـCTR (≥200 انطباع و~0 نقرة) |
| المطوّر (أنت) | **مُنفّذ/مُدمِج فقط** | يطبّق التغيير حين يَعبر أحد الأدلة عتبته — **لا يبادر بالهوى** |

**قاعدة ترجيح التعارض:** الموضوعي (اختبارات، Search Console) **يفوق** الذاتي (تفضيل). الصحّة/السلامة تفوز دائمًا.

### 17.2 الحد الأدنى لتغيير القالب — افتراضُ "الحفاظ" (Preserve-by-default)

> **الافتراض = لا تغيير.** عبء الإثبات على *التغيير* لا على *الإبقاء*. بلا هذا يصبح "التطوّر" بلا حدود.

نقول **"نطوّر القالب"** فقط إذا تحقّق **كل** ما يلي:
1. **مُحفِّز موثّق** مرتبط بمصدر دليل واحد من 17.1.
2. **عبور عتبته الرقمية** (≥2/5 فشل لنفس السبب، أو فرع CTR، أو فشل تعبيري).
3. **يؤثّر على ≥ 2 صفحة** (قاعدة الثلاثة للتعميم) **أو** عيب حاجب على صفحة واحدة.
4. **توثيق:** سبب + دليل + رفع `templateVersion` + إعادة توليد الكل.

نقول **"نُبقيه كما هو"** في كل ما عدا ذلك: رغبة تجميلية، حدس بلا دليل، حالة فردية نادرة، تحسين لـ"صفحة واحدة فقط".

> الخلاصة: **القالب يتغيّر فقط بدليلٍ مُعتَّبٍ يخصّ نوعه ويمسّ ≥2 صفحة — وإلا يبقى.** هذا يحوّل "التطوّر المضبوط" من شعار إلى بوّابة رقمية.
