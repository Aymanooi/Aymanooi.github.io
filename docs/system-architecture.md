# معمارية النظام: مصنع أصول رقمية ينمو تلقائيًا (Digital Asset Factory)

> **التحوّل:** من "موقع أدوات" إلى **نظام** يكتشف الطلب ويحوّله إلى صفحات/أدوات وينشرها ويحسّنها — تلقائيًا، ضمن قيود GitHub Pages.
> **الحالة:** تصميم معماري بانتظار الاعتماد. **لا كود بعد.**

---

## 0) القيد الحاكم والفكرة المحورية

| الحقيقة | الأثر التصميمي |
|---|---|
| GitHub Pages = استضافة **ثابتة فقط** (لا خادم وقت التشغيل) | كل "ذكاء" النظام يحدث **وقت البناء (build-time)**، لا وقت الطلب |
| **GitHub Actions** = محرّك تنفيذ **مجاني** (cron + on-push) للمستودعات العامة | هو "خادم الأتمتة" الذي نملكه دون تكلفة: يكتشف، يولّد، ينشر |
| لا APIs مدفوعة | الاكتشاف يعتمد إشارات **مجانية** (autocomplete، Wikipedia، Search Console) |
| مخرجاتنا صفحات ثابتة | توليد الصفحات = **بيانات → قالب → HTML** بواسطة مولّد (generator) |

**الفكرة المحورية (في جملة):**
> النظام = **خط إنتاج وقت-البناء**: `إشارات طلب مجانية` → `جدول فرص مُسجّل` → `مولّد يحوّل كل فرصة عبر قالب-نمطي إلى صفحة ثابتة` → `نشر تلقائي + SEO مُولَّد + خريطة موقع` — يديره GitHub Actions دوريًا، فيتراكم الأصل بلا تدخّل يدوي.

```
        ┌─────────────────── GitHub Actions (cron + push) ───────────────────┐
        │                                                                     │
 [الطبقة 1]            [الطبقة 2]                    [الطبقة 3]                 │
 اكتشاف الطلب  ──►  تحويل الطلب لصفحات  ──►  نشر + SEO + توزيع                  │
 (إشارات مجانية)     (قالب-نمطي × بيانات)     (generator → static + sitemap)   │
        ▲                                                  │                   │
        └──────────── حلقة تغذية راجعة (Search Console) ◄──┘                   │
        └─────────────────────────────────────────────────────────────────────┘
```

## 1) المعمارية — ثلاث طبقات

### الطبقة 1 — اكتشاف الطلب (Demand Discovery) بلا API مدفوعة

الهدف: تحديد استعلامات عربية **عالية النية ومنخفضة المنافسة** تلقائيًا.

**مصادر الإشارة (كلها مجانية):**

| المصدر | ما يقدّمه | الصدق/المخاطرة |
|---|---|---|
| **Google Autocomplete** (`suggestqueries`) | توسيع البذور العربية إلى ذيل طويل واقعي ("حوّل من ... إلى ...") | غير رسمي → نستخدمه بأدب (rate-limit) ومع بديل يدوي عند الفشل |
| **"عمليات بحث ذات صلة" / People Also Ask** | أسئلة حقيقية يطرحها الناس → صفحات FAQ | استخراج خفيف ومحترم لـ robots |
| **Wikipedia Pageviews API** (رسمي مجاني) | قياس اهتمام نسبي بموضوع | موثوق ومسموح |
| **التوسيع التوافقي (Combinatorial)** | اشتقاق آلاف الفرص: تحويل/حساب بين كل (X,Y) في مجال | قوي جدًا ورخيص — مصدرنا الأساسي |
| **🔁 Google Search Console** (رسمي مجاني) | بعد النشر: استعلامات حقيقية وانطباعات → **الحلقة الذهبية** | الأدق على الإطلاق؛ يغذّي الطبقة 1 |

**التسجيل والترتيب:** كل فرصة تُخزَّن في `data/opportunities.json` بدرجة أولوية:

```
score = demandProxy × feasibilityClientSide × inverseCompetition × arabicGapBonus
```

- `demandProxy`: عمق نتائج autocomplete + اهتمام Wikipedia (تقريبي).
- `feasibilityClientSide`: هل يمكن إنجازها 100% في المتصفح؟ (وإلا تُرفض).
- `inverseCompetition`: تقدير خشن (قوة النطاقات في SERP، ضعف النتائج العربية).
- `arabicGapBonus`: مضاعِف للفرص ذات الطابع العربي (هجري، تفقيط...).

> **صدق:** بلا أدوات SEO مدفوعة، تقدير المنافسة **خشن**. لذا نعتمد إشارتين رخيصتين موثوقتين أكثر — **قابلية التنفيذ + عمق autocomplete** — ونترك التقدير الدقيق لحلقة Search Console بعد النشر.

### الطبقة 2 — تحويل الطلب إلى صفحات (Conversion via Archetypes)

المبدأ الذي يحقق التوسّع: **لا نبني صفحات، بل نبني "أنماطًا" (Archetypes) تولّد صفحات.**

نوعان من الأصول:
- **أدوات تفاعلية** (حساب/تحويل في المتصفح).
- **صفحات معرفية** (إجابات ذيل-طويل لا تحتاج تفاعلًا).

**الأنماط (Archetypes) — كل نمط = قالب واحد + وحدة منطق واحدة + جدول بيانات:**

| النمط | المنطق (في `/lib`) | يولّد صفحات مثل | الكثرة المحتملة |
|---|---|---|---|
| **Converter** (A↔B) | دالة تحويل عامة + جدول وحدات | متر↔قدم، كجم↔رطل، °م↔°ف، صورة PNG↔JPG | عشرات |
| **Calculator** | دالة حساب بارامترية | نسبة مئوية، عمر، ضريبة، قرض | عشرات |
| **Text-Transform** | تحويل نص نقي | عدّاد، حالة الأحرف، تفقيط، إزالة تشكيل | عشرات |
| **Generator** | توليد قيمة | كلمة مرور، QR، UUID، Lorem عربي | عشرات |
| **Lookup/Reference** | بحث في جدول ثابت | كود دولة، رمز عملة، اختصار | مئات |

> **النتيجة الرياضية للتوسّع:** 5 أنماط × (10–40 صفًا في البيانات لكل نمط) = **100+ صفحة** من **5 قوالب فقط**. إضافة صفحة = صفّ بيانات. إضافة فئة كاملة = نمط جديد. **صفر إعادة هيكلة.**

### الطبقة 3 — النشر + SEO + التوزيع (Publishing)

كل ما يلي **مُولَّد آليًا** من البيانات، لا يُكتب يدويًا:

- **المولّد (`scripts/generate.js`)**: `data/*` + `templates/*` → يكتب `tools/<slug>/index.html` لكل فرصة معتمدة.
- **SEO تلقائي لكل صفحة:** `<title>`، `description`، `canonical`، Open Graph، و JSON-LD (`WebApplication` + `BreadcrumbList` + `FAQPage`) — كلها مشتقّة من حقول البيانات.
- **الربط الداخلي تلقائي:** المولّد يبني رسم الروابط (أدوات ذات صلة بنفس النمط/الفئة) → سلطة نطاق + صفحات/جلسة.
- **خريطة الموقع:** `sitemap.xml` + `robots.txt` يُولَّدان من قائمة الصفحات.
- **بطاقات اجتماعية:** صور OG كـ SVG قوالب (ثابتة، صفر تكلفة) لكل صفحة.
- **التوزيع:** ping لخرائط الموقع، RSS للأصول الجديدة، وحلقة Search Console.
- **المحرّك:** GitHub Actions (cron يومي/أسبوعي + عند الدفع) يعيد تشغيل الاكتشاف + التوليد + النشر.

## 2) الـ MVP الصارم (أبسط نسخة تثبت "النظام" لا "الأداة")

المعيار: أصغر شيء يثبت أن **خط الإنتاج يعمل** — أي أن صفحة واحدة تُولَّد من بيانات عبر قالب، فيتضاعف العدد بصفوف بيانات.

**نطاق MVP (نسخة أولى، بلا backend، تكلفة صفر):**

| المكوّن | في MVP | مؤجّل لما بعد |
|---|---|---|
| الأنماط | **نمط واحد فقط: Converter** | بقية الأنماط الأربعة |
| البيانات | **جدول ~10 تحويلات** يدويًا (`data/converters.json`) | اكتشاف آلي |
| الطبقة 1 (اكتشاف) | **v0 يدوي موثّق** + سكربت بسيط يسحب اقتراحات autocomplete إلى ملف | أتمتة كاملة + تسجيل درجات |
| المولّد | `scripts/generate.js` يحوّل 10 صفوف → 10 صفحات + `sitemap.xml` | OG SVG، RSS، رسم روابط متقدم |
| النشر | GitHub Actions: `on push` يولّد ثم ينشر Pages | cron دوري للاكتشاف |
| الجودة | بوّابة جودة يدوية (مراجعة الصفحات الـ10) | بوّابة آلية |

**محصّلة MVP:** أمر واحد (`node scripts/generate.js`) ينتج 10 صفحات حقيقية مفيدة من جدول بيانات + خريطة موقع، وينشرها Actions. **هذا يثبت التوسّع** — لأن الصفحة 11 و100 ليست إلا صفًا إضافيًا.

## 3) آلية التوليد التلقائي للصفحات

### 3.1 خط الأنابيب (Pipeline)

```
data/<archetype>.json   templates/<archetype>.html   /lib/<archetype>.js
        │                        │                          │
        └──────────────┬─────────┘                          │
                       ▼                                     │
              scripts/generate.js  ──(استبدال الرموز + حقن JSON-LD + روابط)──►  tools/<slug>/index.html
                       │                                                         + sitemap.xml + robots.txt
                       ▼
              GitHub Actions  ──►  نشر GitHub Pages
```

### 3.2 القالب الموحّد لكل صفحة (عقد ثابت)

كل صفحة مولّدة تتبع نفس الهيكل (يضمن الاتساق والتوسّع):

```
<head>  → meta فريدة + canonical + OG + JSON-LD (WebApplication+Breadcrumb+FAQ)  [مُولَّد]
nav + breadcrumb (الرئيسية ← الفئة ← الصفحة)                                       [مشترك/مُولَّد]
H1 + جملة فائدة                                                                    [من البيانات]
لوحة الأداة/الإجابة (إدخال → نتيجة فورية)                                          [نمط + بيانات]
كتلة محتوى ≥250 كلمة: كيف تستخدم؟ • حالات • لماذا؟ • FAQ                            [من البيانات]
أدوات ذات صلة (3–4)                                                                [رسم روابط مُولَّد]
footer                                                                            [مشترك]
```

### 3.3 ضمان التوسّع إلى 100+ بلا إعادة هيكلة

| المبدأ | كيف يُفرض |
|---|---|
| **البيانات مصدر الحقيقة الوحيد** | الصفحات **مُولَّدة لا تُحرّر يدويًا** (تُكتب فوقها في كل بناء) |
| **التوليد محدّد ومتكرر (idempotent)** | نفس المدخلات → نفس المخرجات؛ المولّد هو المنتج الوحيد |
| **النمط مفصول عن المثيل** | إضافة صفحة = صف بيانات؛ إضافة فئة = نمط جديد (قالب + وحدة lib) |
| **المنطق معزول ومُختبَر في `/lib`** | كل نمط له اختبارات تمنع الانحدار مع نمو الأعداد |
| **بوّابة جودة آلية** (مرحلة لاحقة) | ترفض الصفحات الرقيقة (محتوى < حد، قيمة مكرّرة) |

## 4) أهداف التصميم الأربعة — كيف يحقّقها النظام

| الهدف | الآلية في هذا التصميم |
|---|---|
| **قابلية التوسّع** | أنماط بارامترية + بيانات → 5 قوالب تنتج 100+ صفحة؛ إضافة = صف |
| **اكتشاف الفرص** | الطبقة 1 (autocomplete + توافقي + Wikipedia + حلقة Search Console) |
| **تحسين SEO تلقائي** | كل وسوم/بيانات منظّمة/روابط داخلية/خريطة تُولَّد من البيانات لكل صفحة |
| **سرعة الإنتاج** | صفحة جديدة = دقائق (صف بيانات)؛ فئة جديدة = ساعات (نمط)؛ كله عبر أمر واحد + CI |

## 5) سجلّ الصدق والمخاطر (Honesty Ledger)

> هذه أهم فقرة. تجاهلها = فشل النظام.

1. **خطر المحتوى الرقيق/البوّابي (Thin/Doorway pages):** توليد صفحات بالجملة دون قيمة فريدة **يعاقبه Google**.
   → **الإلزام:** كل صفحة مولّدة يجب أن تحمل **أداة عاملة فعلًا + محتوى فريدًا**، لا مجرد قالب مكرّر. الكمّ بلا قيمة = ضرر لا نفع. بوّابة الجودة ليست رفاهية.
2. **Google Autocomplete غير رسمي:** قد يتغيّر/يُحجب. → بديل يدوي دائم؛ لا نبني اعتمادًا حرجًا عليه.
3. **تقدير المنافسة خشن** بلا أدوات مدفوعة. → نعوّض بحلقة Search Console الحقيقية بعد النشر.
4. **حصص GitHub Actions:** مجانية للمستودعات العامة ضمن حدود معقولة. → جدولة الاكتشاف **أسبوعيًا** لا كل دقيقة.
5. **SEO يحتاج وقتًا (أشهر).** → مؤشرات الـ30 يومًا = فهرسة + أداء + بدء ظهور، لا إيراد.
